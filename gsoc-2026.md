## 🧭 Proposal Summary

I am José, a Robotics student at the University of Málaga (Spain) and an active contributor to ros2_control. In the past weeks, I have been working on gz_ros2_control, including adding tests (PRs [#765](https://github.com/ros-controls/gz_ros2_control/pull/765) and [#814](https://github.com/ros-controls/gz_ros2_control/pull/814)), exposing and fixing a regression in Rolling (Issue [#801](https://github.com/ros-controls/gz_ros2_control/issues/801), solved with PR [#809](https://github.com/ros-controls/gz_ros2_control/pull/809)) and managing backports across distros (Jazzy merged in PR [#781](https://github.com/ros-controls/gz_ros2_control/pull/781), Kilted and Humble in progress). My contributions have increased test coverage from 5 to 9 demos, with more on the way (the mobile‑robots ones).

This proposal has the goal of building a mission control component for ros2_control: a system that orchestrates controller_manager based on real‑time introspection rather than static presets.

The original project description mentions preset‑based orchestration, but after checking the architecture with mentor Sai Kishor, the expected design is that the mission control should deduce transitions directly from controller_manager introspection without relying on static YAML presets.

Therefore, my proposal is a standalone mission‑control node that orchestrates controller_manager based on real‑time system state, enriched controller status information, and safe transition rules.

I will follow the same principles I have been using so far: upstream integration, reproducibility, and robustness; small, reviewable PRs to ros-controls, CI-backed tests, and a demo showing end‑to‑end orchestration and recovery behavior.

---

## 🔧 Technical Plan

The project will be built incrementally:

1. Status publisher/API in controller_manager and standardized controller status topics.  
2. Dynamic introspection model (without YAML presets).  
3. Mission control node/CLI with safe lifecycle transitions.  
4. Reproducible benchmark, tests, documentation and demo.

- Environment setup

- Agreement with mentors (Sai, Bence)  
- Define the controller status model (pending mentor input)

### **Weeks 3–5**
- Implement a status publisher or equivalent API in controller_manager  
- Add standardized status topics to selected controllers  
- Define the introspection model (without YAML presets)

### **Weeks 6–8**
Implement the mission‑control node/CLI (`ros2_mission_control`):

- reads controller states dynamically  
- orchestrates lifecycle transitions  
- handles retries, timeouts, and rollback  
- optional RViz integration (depending on mentor direction)

### **Weeks 9–10**
- Develop a reproducible benchmark (single robot to multi‑robot)  
- Add robustness tests and visualization hooks

### **Weeks 11–12**
- Documentation  
- CI polish  
- Final PRs  
- Tutorial and demo video

The Mission Control project is exactly the kind of challenge that motivates me. It combines architecture, orchestration, and real‑time reasoning, and it addresses a real gap in the current ros2_control ecosystem.

I am focused on establishing a consistent controller status model and building an orchestration system that improves safety and predictability. My recent contributions, debugging experience, and interest in developer tooling (such as my ros2 debug assistant) prepare me well to deliver a valuable implementation.

I am committed to collaborating closely with the mentors to refine the architecture and create a solution that benefits the whole community.
